lafan1_data_frames: 一个列表，每个元素是一帧的字典，包含各个骨骼关节的位置和方向

BVH文件解析
BVH是BioVision公司推出的一种人体动作捕捉文件格式。这种文件以节点为核心元素，记录连续数帧内人体骨架的运动。BVH文件分为Hierarchy和Motion两部分，Hierarchy部分描述并定义了虚拟角色的树形结构，Motion部分记录每一帧虚拟角色运动的姿态，以lafan1数据集为例：ROOT：根节点，第一个节点通常都是根节点
OFFSET：该节点相对于父节点的偏移，也就是在父节点坐标系下该节点的坐标
CHANNELS：描述并定义该节点的自由度，一般情况下，根节点有6个完整自由度，其他节点有3个旋转自由度
END Site：一个无名节点，表示树形结构的终点
这样的表示方法本质上与机器人DH坐标系表示方法类似，只不过机器人一个关节只有一个旋转自由度，而这里一个节点有三个旋转自由度，再通过正运动学就可以计算出任意状态下各个节点在根节点坐标系下的位置和姿态。

bvh文件解析的主要实现在lafan1.py中的load_bvh_file函数：
首先读取bvh文件原始数据：data = read_bvh(bvh_file)
返回一个 Anim 对象，该对象包含以下数据：

quats: 局部四元数张量，表示每个关节的局部旋转，形状为 (帧数, 关节数, 4)
pos: 局部位置张量，表示每个关节的局部位置，形状为 (帧数, 关节数, 3)
offsets: 关节的局部偏移量，表示骨架结构
parents: 骨骼层次结构，记录每个关节的父节点索引
bones: 骨骼名称列表，如 "Hips", "Spine", "LeftUpLeg" 等
这些数据是从 BVH 文件解析出来的原始运动捕捉数据，其中旋转已经从欧拉角转换为四元数表示。
紧接着进行数据的全局坐标系转化：global_data = utils.quat_fk(data.quats, data.pos, data.parents)
global_data 是一个包含两个元素的元组：

第一个元素: 全局四元数 (帧数, 关节数, 4) - 每个关节在世界坐标系下的绝对旋转方向
第二个元素: 全局位置 (帧数, 关节数, 3) - 每个关节在世界坐标系下的绝对3D坐标位置
这个转换过程通过递归地将每个关节的局部变换累积到其父节点的全局变换上来实现，从而得到每个关节在世界坐标系中的绝对位姿。
最后在for循环中处理每一帧数据，将其转换到适合机器人使用的坐标系中，这里计算的是

位置: 每个骨骼关节在世界坐标系中的3D位置（单位：米）
姿态: 每个骨骼关节在世界坐标系中的旋转方向（四元数表示）
旋转矩阵 [[1, 0, 0], [0, 0, -1], [0, 1, 0]] 实现了坐标系变换（Y-up到Z-up的转换），同时位置从厘米转换为米（除以100）。转换后的坐标系是标准的机器人坐标系（Z轴向上），并且最终返回的是各个frame在机器人坐标系下的绝对位姿。
另外，代码还为左右脚创建了修改版本的姿态，这是因为脚部的姿态需要使用脚趾的方向，而位置使用脚踝的位置，以更好地匹配机器人的脚部结构。

GMR的IK解算
在详细分析GMR的IK解算原理之前，还是先来看看如何将GMR适配到自己的机器人模型中来吧，这是使用GMR最关心的问题了。
根据GMR当前的灵活框架，适配新的机器人模型，除了需要机器人的mujoco配置参数外，只需要新建一份ik_config配置文件就行，配置文件中各参数的含义可在项目中找到，下面重点介绍如何设置参数。
首先就来设置两个ik_match_table中的offset，一般情况下设置旋转就行，先假设人形节点与机器人节点之间没有旋转变换，所以offset=[1,0,0,0]，此时断点运行程序并在显示机器人节点坐标系后暂停(坐标系1)，然后新建另一个mujoco viewer并打开机器人模型，将机器人各关节构型设置(坐标系2)为与程序断点运行时显示的构型一致，此时对比两边坐标系的差距，从坐标系1旋转至坐标系2就是该节点的旋转ofset值。


以LeftArm这个节点为例，从左边的坐标系1到右边的坐标系2，XYZ欧拉角为[90，-90，0]，对应的四元数为[0.5,0.5,-0.5,-0.5]，该值与当前g1的ik_config配置参数值一样，说明方法的有效性。
推荐两个在线四元数转化与计算的网站：四元数转换，四元数计算
按照相同方法设置完其他节点的offset后，根据不同机器人体型差异来设置human_scale_table，最终要实现的效果就是坐标系1和坐标系2能够完全同步（但在不设置位置offset时是几乎不可能的，所以位置同步的容忍度会更高）。
最后再来调节位置约束和姿态约束的权重系数，对于明显无法位置同步的节点，可以直接将位置权重设为零。
源码中关于IK解算的主要代码在retarget函数中，第一步将人类动捕数据成比例的映射至机器人对应的frame，获得机器人所需的target_frame，然后利用mink，一步步迭代求解出关节角度。具体的IK求解策略和流程其实很简单，就是利用mink.FrameTask，设置需要对齐的frame，相应的位置和姿态权重，然后将target_frame的位姿设置为对应task的目标位姿，最后循环迭代求解就好了。

总结
GMR在运动重定向方面还是具有很强的通用性的，覆盖了主流的动捕源数据与机器人模型，能够快速生成并输出机器人重定向后的运动轨迹，对于复杂的运动轨迹，如果想要作为后续模型训练的输入，还是需要疯狂调参再手动微调的。
